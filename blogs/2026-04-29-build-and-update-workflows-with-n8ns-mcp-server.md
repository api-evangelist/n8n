Now you can build new ones from scratch and update existing ones, directly in your n8n instance.</p><ul><li><strong>Go from prompt to a ready-to-run workflow.</strong> <br />Tell your AI client what you want. No messing with JSON files or copy-pasting errors.</li><li><strong>Works in whatever AI client you already use.</strong> <br />Claude, ChatGPT, Cursor, Windsurf - if it &#x201c;speaks&#x201d; MCP, you can point it at n8n. No new tool to learn, no context-switching.</li><li><strong>First-party, native, and available to all.</strong> <br />Built into every edition of n8n: Cloud, Enterprise, and the free self-hosted Community Edition. Maintained by n8n, no third-party service to run alongside your instance.</li></ul><p>It&apos;s been in public preview for the past few weeks and the n8n team already uses it daily. We can&apos;t wait for you to try it out.</p><p><em>Note: This article is about the MCP server built into n8n, not the MCP Server Trigger node. The former lets external AI clients connect to your entire instance. The latter exposes a single workflow as an MCP server.</em></p><h2 id="using-it-a-real-example"><strong>Using it: a real example</strong></h2><p>Here&#x2019;s what it looks like end-to-end, with a simple workflow I built to test it out, using Claude Desktop (Chat) and Opus 4.6.</p><p>Just tell it what you want it to build:</p><blockquote><em>&quot;I want you to create an n8n workflow that once a day at 7am sends me an email with today&apos;s forecast. It also has tools for&#xa0;<strong>validating workflows, running test executions</strong>, and <strong>generating test data</strong>.</p><p>So when your MCP client creates a workflow, the flow usually looks like this:</p><ol><li>Generate the workflow</li><li>Validate it (catches most errors before execution)</li><li>If validation fails, fix it and re-validate</li><li>Execute it, generating test data if needed</li><li>If execution fails, read the error, fix the workflow, run it again</li></ol><p>All of that happens inside your MCP client while you watch.</p><p><strong>You don&apos;t need to do anything.</strong></p><p>No copy-paste, no pasting errors back, no debugging handoffs. If your prompt was specific enough, you end up with a working workflow, and if the first attempt had issues, the model iterates on its own to fix it.</p><p><em>Please note: The process described above is not part of any instructions given to the client. The model is not told to test the workflow after the build process, it&#x2019;s just something most clients and LLMs naturally do.</em></p><div class="kg-card kg-callout-card kg-callout-card-blue"><div class="kg-callout-emoji">&#x1f6e0;&#xfe0f;</div><div class="kg-callout-text">We&apos;re shipping this as a <b><strong style="white-space: pre-wrap;">Public Preview.</strong></b><br />The n8n team already uses it daily for real work, but complex workflows often need a second (or third) pass, and we&apos;re still smoothing rough edges.</div></div><h2 id="connecting-your-mcp-client"><strong>Connecting your MCP client</strong></h2><p>Connecting to the n8n MCP server works the same way as connecting to any other MCP server, though you need to first enable it for your n8n instance.</p><ol><li>Enable the MCP server</li><li>Copy the connection details</li><li>Add the n8n MCP server to your MCP client of choice</li><li>Restart or reload the client</li></ol><p>The setup guide also includes per-client instructions for Claude Desktop, Claude Code, Codex CLI, and Google ADK.</p><p><strong>&#x2192; Check out the </strong><a href="https://docs.n8n.io/advanced-ai/mcp/accessing-n8n-mcp-server/"><strong>Full Setup Guide</strong></a><strong> for details.</strong></p><div class="kg-card kg-callout-card kg-callout-card-blue"><div class="kg-callout-emoji">&#x1f64f;</div><div class="kg-callout-text">Please use n8n version <code style="white-space: pre-wrap;">2.18.4</code> or higher for the best experience creating workflows with the MCP server.</div></div><p><strong>Alternatively, check out our video which covers the MCP setup and some additional tips.</strong></p><figure class="kg-card kg-embed-card kg-card-hascaption"><figcaption><p dir="ltr"><span style="white-space: pre-wrap;">Official Announcement for the n8n MCP Server Update to Create and Update Workflows</span></p></figcaption></figure><h2 id="getting-the-most-out-of-it"><strong>Getting the most out of it</strong></h2><p>Here are a few things we&apos;ve learned so far. Some of these are tips, some are known rough edges that we&#x2019;re working on.</p><p><strong>Tips</strong></p><ul><li><strong>Use a Coding Agent</strong><br />Our internal testing showed Claude Code gets better results than Claude Chat using the same prompt and the same models. </li><li><strong>Tell it how to build, not just what to build</strong><ul><li>If you&#x2019;re just getting started with n8n, you can skip this tip and just describe the problem you&#x2019;re trying to solve in plain english.</li><li>On the other hand, if you&#x2019;re a veteran n8n builder, be explicit in what nodes to use. For example, whether it uses Code nodes or native ones, whether formatting lives in a template or a script, or whether the timezone is hardcoded.</li></ul></li><li><strong>Name the services you want</strong> <br />When two services could plausibly do the job, like HTTP Request vs. a dedicated integration, or Gmail vs. a generic SMTP node, tell it which one to use. The model will guess otherwise, and it doesn&apos;t always guess well.</li><li><strong>Iterate, don&apos;t restart</strong> <br />If it got 80% right, refine in the same conversation. Starting over usually loses context and makes things worse.</li><li><strong>If you make any changes in the UI, tell it to check the latest version</strong> This might be obvious, but a few times I made changes mid-session via the UI and forgot to tell Claude to check for my changes before continuing. Those changes were over-written.</li><li><strong>Ask it what it got wrong</strong> <br />After it builds, ask &quot;what would have been better if I&apos;d mentioned it upfront?&quot; The answer is often one sentence that&apos;ll save you a rebuild next time.</li></ul><p><strong>What it can get wrong</strong></p><ul><li><strong>Silent design choices on the first pass</strong> <br />The model makes small decisions it doesn&apos;t always surface. If something feels off, it usually is. Ask what it decided and don&apos;t assume nothing was decided.</li><li><strong>Over-engineering before you push back</strong> <br />First attempts are sometimes more complex than needed, like a 50-line Code node where a template and a Set node would do. A quick &quot;this feels too code-heavy&quot; usually produces a simpler second version.</li><li><strong>Complex branching.</strong> <br />Workflows with a lot of conditional paths or nested logic often need manual cleanup. We&apos;re actively working on this.</li><li><strong>Node selection when options overlap.</strong> <br />If multiple nodes could do the job, it sometimes picks the wrong one. This is the single most common thing worth steering explicitly.</li><li><strong>Platform quirks discovered mid-build.</strong> <br />The model learns some of the SDK&apos;s constraints the hard way, by hitting a blocked method, a credential it can&apos;t bind, a node that can&apos;t self-reference. A retry cycle or two is normal on non-trivial workflows. That&apos;s the guardrails working, not the model failing.</li></ul><p>We&apos;re fixing rough edges quickly. Expect things to get noticeably better over the next few releases.</p><h2 id="a-quick-word-on-how-its-built"><strong>A quick word on how it&apos;s built</strong></h2><p>The MCP server generates a TypeScript representation of the workflow rather than raw JSON. In practice, that means the model has to produce something that type-checks and compiles before anything touches your instance, leading to a more reliable solution.</p><p>A few other things worth knowing about the native MCP server:</p><ul><li><strong>No extra service to run or maintain</strong><br />It&apos;s part of n8n. If your instance is up, the MCP server is up.</li><li><strong>Deeper integration</strong><br />Because it&apos;s built into n8n, it can use capabilities that aren&apos;t exposed through the public API.</li><li><strong>Generates code, not JSON</strong><br />The TypeScript approach (above) produces much better results than direct JSON generation in our testing.</li></ul><p>&#x1f449; A full technical deep-dive on the architecture is coming soon!</p><p>We also want to acknowledge the community members that built their own open-source MCP servers to create workflows. The people behind them are often among our most engaged users and we appreciate the hard work that went into those solutions.</p><h2 id="start-building-today"><strong>Start building today!</strong></h2><p>So go build something.
